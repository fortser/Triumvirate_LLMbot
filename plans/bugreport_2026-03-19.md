## BUG-1: Массовые 503 Service Unavailable от OpenRouter

**Severity:** Critical
**Затронутые модели:** 7 из 9 (все кроме openai/gpt-5-mini и openai/gpt-4.1-nano)
**Масштаб:** 2370 ошибок 503 за день

### Описание

Подавляющее большинство LLM-запросов к OpenRouter возвращают `503 Service Unavailable`. Модели физически недоступны — не проблема парсинга или промптов. Бот делает 3 попытки (retry), все 3 получают 503, после чего делает случайный ход.

### Затронутые модели и масштаб

| Модель | 503 ошибок | Всего LLM-вызовов | % ошибок |
|--------|:---:|:---:|:---:|
| arcee-ai/trinity-large-preview:free | 801 | ~860 | 93% |
| openrouter/hunter-alpha | 528 | ~972 | 54% |
| meta-llama/llama-3.3-70b-instruct:free | 438 | ~751 | 58% |
| stepfun/step-3.5-flash:free | 198 | ~224 | 88% |
| minimax/minimax-m2.5:free | 165 | 165 | 100% |
| openrouter/healer-alpha | 144 | ~1059 | 14% |
| z-ai/glm-4.5-air:free | 96 | ~145 | 66% |

### Причина

OpenRouter free-tier модели имеют нестабильную доступность. Проблема на стороне провайдера, не бота. В исторических данных (до 19.03) ошибок 503 **вообще не было** — это новая проблема, начавшаяся сегодня.

### Рекомендация

- [ ] Добавить в multi_bot.py проверку доступности модели перед запуском игры (health check через OpenRouter API `/api/v1/models`)
- [ ] Добавить circuit breaker: после N последовательных 503 — прекращать игру, не тратить ходы на рандом
- [ ] Логировать причину 503 (overloaded / maintenance / rate limit) если OpenRouter возвращает тело ответа
- [ ] Рассмотреть fallback на другую модель при недоступности основной

## BUG-2: healer-alpha снят с API (404 Not Found)

**Severity:** Critical
**Затронутая модель:** openrouter/healer-alpha
**Масштаб:** 915 ошибок 404, 353 хода случайных, 5 игр бесполезных

### Описание

Модель `openrouter/healer-alpha` возвращает 404 с сообщением:

```
Healer Alpha was a stealth model revealed on March 18th as an early testing
version of MiMo-V2-Omni. Find it here: https://openrouter.ai/xiaomi/mimo-v2-omni
```

Модель была "стелс-моделью", раскрытой 18 марта. API endpoint удалён, но бот продолжает отправлять запросы и получать 404 на каждый ход.

### Причина

`models_pool.json` содержит устаревший идентификатор `openrouter/healer-alpha`, который больше не существует на OpenRouter.

### Рекомендация

- [ ] Удалить `openrouter/healer-alpha` из `models_pool.json`
- [ ] Заменить на `xiaomi/mimo-v2-omni` (если модель доступна) или удалить
- [ ] Аналогично проверить `openrouter/hunter-alpha` — может быть тоже стелс-моделью
- [ ] Добавить валидацию: перед запуском игры проверить, что модель существует на OpenRouter
- [ ] Добавить обработку 404 в `llm_client.py` — не делать retry при 404 (бессмысленно)

## BUG-3: Модели галлюцинируют нелегальные ходы в TRIUMVIRATE нотации

**Severity:** High
**Затронутые модели:** arcee-ai/trinity-large-preview:free, openai/gpt-4.1-nano
**Масштаб:** Парсинг находит координаты, но ход нелегален

### Описание

При использовании TRIUMVIRATE нотации (`use_triumvirate_notation: True`) модели генерируют JSON с корректным синтаксисом, но нелегальными ходами. Парсер находит координаты, конвертирует в серверную нотацию, но ход отсутствует в `legal_moves`.

### Примеры

**arcee-ai/trinity, Game 0aa2d2e1, Move 2:**
- LLM ответ: `{"move_from": "W3/B3.2", "move_to": "C/W.B"}` (слон C1 на D4)
- Реальность: слон на C1 может пойти на D2, E3, F4, G9, H10 — но **не D4**
- D4 (C/W.B) — центральная розетка, визуально привлекательная, но не на диагонали слона
- Модель "видит" хороший ход в нотации, но не проверяет, может ли фигура туда пойти
- Три попытки LLM — все три с нелегальными ходами → fallback random

**openai/gpt-4.1-nano, Game 5ec1f069, Move 3:**
- Ответ 1: `W3/B3.3` → `W2/B2.2` (D1→C2) — нелегальный для короля в шахе
- Ответ 2: `W3/R3.3` → `W1/B1.2` (E1→C3) — нелегальный для короля
- Ответ 3: `W3/B3.3` → `C/W.B` (D1→D4) — нелегальный
- Все три попытки нелегальны — модель не видит, куда King реально может уйти из-под шаха

### Причина

1. **TRIUMVIRATE нотация сложна для LLM.** Радиально-кольцевая система (W3/B3.2) менее интуитивна, чем серверная (C1). Модели хуже понимают геометрию доски в этой нотации.
2. **Модели не проверяют легальность.** Промпт содержит список легальных ходов, но модели их игнорируют и "выдумывают" ход на основе позиционного анализа.
3. **Последняя попытка retry не упрощает формат.** На последнем retry бот должен показывать прямой список легальных ходов, но непонятно, работает ли это.

### Рекомендация

- [ ] Добавить в промпт явное требование: "ONLY use moves from the Legal Moves list. Do NOT invent moves."
- [ ] На последнем retry показывать ТОЛЬКО список легальных ходов без описания позиции, чтобы модель выбирала из списка
- [ ] Рассмотреть: для слабых моделей (free-tier) использовать серверную нотацию вместо TRIUMVIRATE — она проще для LLM
- [ ] Добавить метрику `illegal_move_rate` в трейсы — процент ходов, где модель предложила нелегальный ход

## BUG-4: Бот продолжает играть при 100% fallback rate

**Severity:** Medium
**Затронутые модели:** все модели с 0% success rate
**Масштаб:** 6 моделей отыграли 1088 ходов рандомом без единого осознанного хода

### Описание

Когда модель недоступна (503/404) или не может выдать легальный ход, бот делает случайный ход и продолжает игру. Нет механизма остановки при систематических ошибках. В результате:

- `openrouter/healer-alpha` — 353 случайных хода, 0 осознанных, 5 игр
- `openrouter/hunter-alpha` — 324 случайных хода, 0 осознанных, 5 игр
- `meta-llama/llama-3.3-70b-instruct:free` — 245 случайных ходов, 0 осознанных, 4 игры

Эти игры бессмысленны — они не дают данных о модели и засоряют логи.

### Причина

В `bot_runner.py` нет проверки "сколько ходов подряд были fallback". Бот безусловно продолжает игру до её конца на сервере.

### Рекомендация

- [ ] Добавить параметр `max_consecutive_fallbacks` (default: 10) — после N рандомных ходов подряд бот сдаётся (`resign()`) и логирует причину
- [ ] Добавить параметр `min_success_rate` (default: 0.1) — если после 20 ходов success rate < 10%, бот сдаётся
- [ ] Добавить предупреждение в GUI: "Model has failed N consecutive moves, consider stopping"
- [ ] При 100% 503/404 — сдаваться немедленно после 3 хода (нет смысла ждать)

## BUG-5: 429 Rate Limit при массовом запуске free-tier моделей

**Severity:** Medium
**Затронутые модели:** meta-llama/llama-3.3-70b-instruct:free (311 ошибок), z-ai/glm-4.5-air:free (34 ошибки)
**Масштаб:** 345 ошибок 429

### Описание

Free-tier модели на OpenRouter имеют строгие rate limits. При запуске нескольких ботов одновременно (multi-bot режим с `--bots 5+`) один аккаунт быстро упирается в лимит. Модель чередует 503 и 429 — иногда доступна, но rate-limited.

### Дополнительно

У `meta-llama/llama-3.3-70b-instruct:free` обнаружены 2 ошибки `402 Payment Required` — возможно, аккаунт исчерпал бесплатный лимит или модель перешла на платный tier.

### Причина

1. Все боты используют один API-ключ OpenRouter
2. Free-tier имеет лимит ~20 req/min (зависит от модели)
3. `--start-delay` помогает на старте, но не при параллельной игре

### Рекомендация

- [ ] Добавить adaptive rate limiting в `llm_client.py`: при получении 429 — ждать `Retry-After` секунд (если заголовок есть) или экспоненциальный backoff
- [ ] Для free-tier моделей автоматически увеличивать `--start-delay` до 30+ секунд
- [ ] Ограничить количество параллельных free-tier ботов (max 2-3 на один аккаунт)
- [ ] Добавить учёт rate limits per model (разные модели имеют разные лимиты)
- [ ] Обработать 402 — если аккаунт исчерпан, не делать retry

## BUG-6: prompt_pipeline не сохраняет response_format и notation_mode в трейс

**Severity:** Low
**Затронуто:** все трейсы

### Описание

В трейс-файлах поле `prompt_pipeline` не содержит:
- `response_format` — какой формат ответа использовался (json / json_thinking / simple)
- `notation_mode` — какая нотация использовалась (server / triumvirate)

Вместо этого используется `use_triumvirate_notation: True/False` и формат ответа можно косвенно определить по `response_format_instruction`, но это неудобно для анализа.

### Причина

`tracer.py` записывает `prompt_pipeline` как есть из `prompt_builder.py`, но тот не добавляет metadata о настройках.

### Рекомендация

- [ ] Добавить в `prompt_pipeline` явные поля: `response_format: "json_thinking"`, `notation_mode: "triumvirate"`
- [ ] Сохранять полную конфигурацию бота в первом трейсе каждой игры (модель, провайдер, все параметры)
- [ ] Это упростит автоматический анализ: "какой формат лучше для какой модели"

## Итог

**87% ходов сегодня — случайные.** Главная причина — массовая недоступность OpenRouter free-tier (BUG-1) и удалённая модель healer-alpha (BUG-2). Только `openai/gpt-5-mini` (100% success) и частично `openai/gpt-4.1-nano` (86%) реально играли. Остальные 7 моделей генерировали бессмысленные данные.

Для следующих запусков необходимо:
1. Удалить мёртвые модели из пула
2. Добавить circuit breaker (автоматическая остановка при системных ошибках)
3. Использовать платные модели или проверять доступность free-tier перед стартом
