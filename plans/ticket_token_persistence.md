# TICKET: Персистентное хранение токенов игр для graceful resign

**Статус:** Open
**Приоритет:** Medium
**Тип:** Feature
**Дата:** 2026-03-19

## Проблема

При аварийном закрытии бота (Ctrl+C, крэш, закрытие терминала) активные игры остаются на сервере в статусе `playing`. Без `player_token` невозможно выполнить resign — игры висят до серверного таймаута, блокируя слоты (лимит ~10 игр).

**Воспроизведение:**
1. Запустить `--headless --bots 5`
2. Закрыть терминал до завершения игр
3. Запустить новых ботов → `503 Service Unavailable` на `/api/v1/join`

## Решение

Сохранять активные сессии (token + game_id + metadata) в файл. При запуске — проверять файл и resign по сохранённым токенам.

## Требования

### 1. Файл сессий `logs/active_sessions.json`

```json
[
  {
    "game_id": "5ec1f069-...",
    "player_token": "eyJ...",
    "color": "white",
    "bot_name": "LLM_OpenRouter_openai/gpt-4.1-nano",
    "model": "openai/gpt-4.1-nano",
    "joined_at": "2026-03-19T12:38:43Z"
  }
]
```

### 2. Жизненный цикл записей

| Событие | Действие |
|---------|----------|
| `arena_client.join()` успешен | Добавить запись в файл |
| Игра завершена (finished/eliminated/resigned) | Удалить запись из файла |
| Запуск бота | Проверить файл, resign по всем сохранённым токенам |

### 3. CLI-команда для ручной очистки

```bash
python main.py --cleanup
```

- Читает `active_sessions.json`
- Делает resign по каждому токену
- Выводит результат (успех/ошибка на каждую сессию)
- Очищает файл

### 4. Автоматический cleanup при старте

При запуске бота (до `join`):
1. Прочитать `active_sessions.json`
2. Для каждой записи попытаться resign
3. Логировать результат (игра могла уже завершиться — это ок)
4. Очистить файл

## Затрагиваемые модули

| Модуль | Изменения |
|--------|-----------|
| `arena_client.py` | Добавить методы `save_session()`, `load_sessions()`, `cleanup_sessions()` |
| `bot_runner.py` | Вызов `save_session()` после join, удаление после завершения |
| `multi_bot.py` | Cleanup перед запуском пула ботов |
| `main.py` | Аргумент `--cleanup`, вызов cleanup при старте |

## Граничные случаи

- **Конкурентная запись:** несколько ботов пишут в один файл → использовать file lock (`fcntl`/`msvcrt`) или per-bot файлы
- **Битый JSON:** обернуть чтение в try/except, при ошибке — переименовать в `.bak` и начать заново
- **Токен истёк на сервере:** resign вернёт ошибку — логировать и удалить запись
- **Файл отсутствует:** создавать при первой записи, отсутствие = нет активных сессий

## Оценка объёма

- ~50 строк в `arena_client.py` (persistence logic)
- ~15 строк в `bot_runner.py` (вызовы save/remove)
- ~10 строк в `multi_bot.py` (cleanup при старте)
- ~10 строк в `main.py` (аргумент --cleanup)

## Альтернативы (отклонены)

1. **Signal handler (SIGINT/SIGTERM):** ненадёжен на Windows, не спасает от kill -9 / закрытия терминала
2. **Хранение в SQLite:** избыточно для ~10 записей
3. **Per-bot отдельные файлы:** проще concurrency, но сложнее cleanup — можно рассмотреть если file lock окажется проблемой
