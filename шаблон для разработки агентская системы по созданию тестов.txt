# Writing Tests

**Core principle:** Test user-observable behavior with real dependencies.
Tests should survive refactoring.

> "The more your tests resemble the way your software is used, 
> the more confidence they can give you." — Kent C. Dodds

## Testing Trophy Model

| Priority | Type        | When                                            |
|----------|-------------|-------------------------------------------------|
| 1st      | Integration | Default — multiple units with real dependencies |
| 2nd      | E2E         | Complete user workflows                         |
| 3rd      | Unit        | Pure functions only (no dependencies)           |

## Mocking Guidelines

**Default: Don't mock. Use real dependencies.**

**Only mock:**
- External HTTP/API calls (внешние сервисы)
- Time/randomness (datetime.now, random, uuid)
- Third-party services (оплата, email, SMS, push)
- Файловая система (только если I/O-heavy)

**Never mock:**
- Internal modules (свои модули проекта)
- Database queries (использовать тестовую БД)
- Business logic (бизнес-логику)
- Your own code calling your own code

**Before mocking, ask:** "Какие побочные эффекты у этого? Нужны ли 
они моему тесту?" Если не уверен — сначала запусти с реальной 
зависимостью, потом добавь минимальный мок только где необходимо.

## Test Type Decision

```
Полный пользовательский workflow?     → E2E test
Чистая функция (без side effects)?    → Unit test
Всё остальное                         → Integration test
```

## Assertion Strategy

| Context   | Assert On                    | Avoid                        |
|-----------|------------------------------|------------------------------|
| API       | Response body, status code   | Internal DB state            |
| Library   | Return values, exceptions    | Private methods              |
| Server    | Client-visible events/data   | Internal queues/buffers      |
| CLI       | Exit code, stdout/stderr     | Internal variables           |

## Anti-Patterns

| Pattern                              | Fix                              |
|--------------------------------------|----------------------------------|
| Testing mock calls                   | Test actual outcome              |
| Test-only methods in production code | Move to test utilities           |
| `time.sleep(N)`                      | Use condition-based waiting      |
| Asserting on internal state          | Assert on observable output      |
| Incomplete/stale mocks              | Mirror real API completely       |
| 500-line test function               | Split into focused tests         |
| Copy-pasted tests with tiny diffs    | Use @pytest.mark.parametrize     |

## Quality Checklist

- [ ] Happy path covered
- [ ] Error conditions handled  
- [ ] Boundary/edge cases included
- [ ] Real dependencies used (minimal mocking)
- [ ] Tests survive renaming internal fields/methods
- [ ] Test names describe behavior, not implementation
- [ ] No test depends on execution order of other tests

## Language-Specific Patterns
- **Python**: See [references/python.md](references/python.md)

---
**Remember:** Behavior over implementation. Real over mocked. 
Outputs over internals.

# Python Testing Patterns

## Pytest Configuration (pyproject.toml)

```toml
[tool.pytest.ini_options]
addopts = "-ra -q --cov={{PACKAGE}} --cov-report=term-missing --strict-markers"
testpaths = ["tests"]
asyncio_mode = "auto"
markers = [
    "slow: long-running tests",
    "integration: integration tests with real dependencies",
    "e2e: end-to-end tests",
]
```

## Fixture Patterns

```python
import pytest

# --- Scope: function (default) — чистая изоляция каждого теста ---
@pytest.fixture
def db_session(db_engine):
    """Транзакция с автооткатом — каждый тест получает чистую БД."""
    connection = db_engine.connect()
    transaction = connection.begin()
    session = Session(bind=connection)
    yield session
    session.close()
    transaction.rollback()
    connection.close()

# --- Scope: session — дорогие ресурсы создаются один раз ---
@pytest.fixture(scope="session")
def db_engine():
    engine = create_engine(TEST_DATABASE_URL)
    Base.metadata.create_all(engine)
    yield engine
    engine.dispose()

# --- Factory fixture — когда нужно создавать объекты с вариациями ---
@pytest.fixture
def make_user(db_session):
    def _make(name="test", email=None, **kwargs):
        user = User(name=name, email=email or f"{name}@test.com", **kwargs)
        db_session.add(user)
        db_session.flush()
        return user
    return _make
```

## Data Generation (Polyfactory)

```python
from polyfactory.factories.pydantic_factory import ModelFactory

class UserFactory(ModelFactory[User]):
    __model__ = User
    is_active = True

def test_admin_access(client):
    admin = UserFactory.build(role=Role.ADMIN)
    response = client.post("/login", json=admin.model_dump())
    assert response.status_code == 200
```

## Mocking External Boundaries Only (respx + httpx)

```python
import respx, httpx

@respx.mock
async def test_external_api():
    respx.get("https://api.external.com/data").mock(
        return_value=httpx.Response(200, json={"result": "ok"})
    )
    result = await my_service.fetch_external_data()
    assert result == "ok"
```

## Async Testing (FastAPI / aiohttp / etc.)

```python
from httpx import AsyncClient, ASGITransport

@pytest.fixture
async def client(db_session):
    app.dependency_overrides[get_db] = lambda: db_session
    transport = ASGITransport(app=app)
    async with AsyncClient(transport=transport, base_url="http://test") as c:
        yield c
    app.dependency_overrides.clear()

async def test_create_resource(client):
    response = await client.post("/resources", json={"name": "test"})
    assert response.status_code == 201
```

## Integration Testing (Testcontainers)

```python
from testcontainers.postgres import PostgresContainer

@pytest.fixture(scope="session")
def postgres():
    with PostgresContainer("postgres:16-alpine") as pg:
        yield pg

@pytest.fixture(scope="session")  
def db_engine(postgres):
    engine = create_engine(postgres.get_connection_url())
    Base.metadata.create_all(engine)
    yield engine
    engine.dispose()
```

## Parametrize

```python
@pytest.mark.parametrize("input_val,expected", [
    pytest.param("valid@email.com", True, id="valid_email"),
    pytest.param("no-at-sign", False, id="missing_at"),
    pytest.param("", False, id="empty_string"),
])
def test_email_validation(input_val, expected):
    assert validate_email(input_val) == expected
```

## Property-Based Testing (Hypothesis)

```python
from hypothesis import given, strategies as st

@given(st.integers(), st.integers())
def test_addition_commutative(a, b):
    assert add(a, b) == add(b, a)
```

# Property-Based Testing Guide

## When to Invoke (Automatic Detection)

Invoke when you detect:

- **Serialization pairs**: encode/decode, serialize/deserialize, 
  to_json/from_json, to_dict/from_dict, dump/load
- **Parsers**: URL parsing, config parsing, protocol parsing
- **Normalization**: normalize, sanitize, clean, canonicalize
- **Validators**: is_valid, validate, check_*
- **Data structures**: Custom collections with add/remove/get
- **Mathematical/algorithmic**: Pure functions, sorting, comparators
- **State machines**: apply/undo, transitions with invariants

## Property Catalog

| Property         | Formula                         | Example Use Case                        |
|-----------------|---------------------------------|-----------------------------------------|
| **Roundtrip**    | `decode(encode(x)) == x`       | Any serialize/deserialize pair          |
| **Idempotence**  | `f(f(x)) == f(x)`              | Normalization, formatting, dedup        |
| **Invariant**    | Property holds before/after     | Size constraints, type preservation     |
| **Commutativity**| `f(a, b) == f(b, a)`           | Distance, set operations                |
| **Associativity**| `f(f(a,b),c) == f(a,f(b,c))`  | Merge operations                        |
| **Inverse**      | `f(g(x)) == x`                 | encrypt/decrypt, compress/decompress    |
| **Oracle**       | `new_impl(x) == reference(x)`  | Optimized code vs brute-force           |
| **Easy to Verify**| Verify output property         | `is_sorted(sort(x))`                   |
| **No Exception** | No crash on valid input         | Baseline smoke test                     |

**Strength:** No Exception → Invariant → Idempotence → Roundtrip

## Core Patterns

### Basic

```python
from hypothesis import given, strategies as st, settings, assume

@given(st.text(), st.text())
def test_concat_length(a, b):
    assert len(a + b) == len(a) + len(b)
```

### Custom Composite Strategy

```python
@st.composite
def valid_domain_objects(draw):
    """{{ЗАПОЛНИТЬ: генерация валидного объекта вашего домена}}"""
    name = draw(st.text(min_size=1, max_size=100))
    value = draw(st.integers(min_value=0, max_value=10000))
    assume(is_valid_combination(name, value))
    return DomainObject(name=name, value=value)
```

### Roundtrip

```python
@given(valid_domain_objects())
def test_serialization_roundtrip(obj):
    serialized = obj.serialize()
    restored = DomainObject.deserialize(serialized)
    assert restored == obj
```

### Invariant

```python
@given(valid_domain_objects(), st.data())
def test_operation_preserves_invariant(obj, data):
    action = data.draw(st.sampled_from(obj.available_actions()))
    result = obj.apply(action)
    assert result.is_valid(), f"Invariant broken after {action}"
```

### Oracle (optimization verification)

```python
@given(valid_domain_objects())
@settings(max_examples=200)
def test_optimized_matches_reference(obj):
    fast = optimized_algorithm(obj)
    slow = reference_algorithm(obj)
    assert fast == slow
```

### Stateful Testing

```python
from hypothesis.stateful import RuleBasedStateMachine, rule, precondition

class SystemStateMachine(RuleBasedStateMachine):
    """{{ЗАПОЛНИТЬ: модель вашей системы}}"""
    
    def __init__(self):
        super().__init__()
        self.system = System.create()
    
    @rule(data=st.data())
    def perform_action(self, data):
        actions = self.system.available_actions()
        if actions:
            action = data.draw(st.sampled_from(actions))
            self.system.apply(action)
            assert self.system.is_valid()

TestSystem = SystemStateMachine.TestCase
```

### CI Profiles

```python
from hypothesis import settings, Phase

settings.register_profile("ci", max_examples=500, deadline=None)
settings.register_profile("dev", max_examples=50)
settings.register_profile("debug", max_examples=10,
                          phases=[Phase.explicit, Phase.generate])
# Использование: pytest --hypothesis-profile=ci
```

# {{PROJECT_NAME}} — Test Suite Generator

## Project Context

{{ЗАПОЛНИТЬ: 5-10 строк описания проекта. Что он делает, какая архитектура,
ключевые модули, основные внешние зависимости. Чем конкретнее — тем лучше
агент поймёт, ЧТО тестировать.}}

**Основной пакет:** `src/{{PACKAGE}}/`
**Фреймворк:** {{FastAPI / Django / Flask / asyncio / etc.}}
**БД:** {{PostgreSQL / Redis / MongoDB / SQLite / etc.}}
**Ключевые паттерны:** {{DDD / CQRS / Event Sourcing / MVC / etc.}}

## Two-Step Process

### ШАГ 1: Идентификация сценариев (ПЛАНИРОВАНИЕ)

**Цель:** Составить ПОЛНЫЙ список всех тестовых сценариев ДО написания кода.

Для КАЖДОГО модуля в `src/{{PACKAGE}}/` выполни анализ:

1. **Публичный интерфейс** — перечисли все публичные функции/методы/классы
2. **Happy path** — основной сценарий использования каждой функции
3. **Edge cases** — граничные значения, пустые коллекции, максимумы
4. **Error paths** — невалидные входы, таймауты, недоступные ресурсы
5. **Инварианты** (используй Skill `property-based-testing`):
   {{ЗАПОЛНИТЬ: перечислить 5-10 ключевых инвариантов вашего домена.
   Примеры:
   - "Баланс счёта никогда не становится отрицательным"
   - "Сериализация/десериализация состояния идемпотентна"
   - "Оптимизированный алгоритм даёт тот же результат, что и эталонный"
   - "После любой валидной операции система остаётся в консистентном состоянии"
   }}
6. **Concurrency** (если есть async/многопоточность) — гонки, deadlocks
7. **State transitions** — жизненный цикл основных сущностей

**Формат результата:** Сохрани в `plans/test-scenarios.md` используя шаблон
из [references/scenarios-template.md](references/scenarios-template.md)

**⚠️ НЕ ПИШИ КОД ТЕСТОВ НА ШАГЕ 1. Только список сценариев.**

### ШАГ 2: Реализация тестов (ИМПЛЕМЕНТАЦИЯ)

**После утверждения списка сценариев** реализуй тесты:

- Skill(`writing-tests`) — философия: поведение > реализация
- Skill(`pytest-patterns`) — механика: fixtures, parametrize, async
- Skill(`property-based-testing`) — инварианты, roundtrip, oracle

**Порядок реализации:**

```
1. tests/conftest.py           — общие fixtures
2. tests/unit/                 — чистые функции без зависимостей
3. tests/integration/          — мульти-модульные с реальными зависимостями
4. tests/property/             — hypothesis-based property tests
5. tests/e2e/                  — полные пользовательские сценарии
```

**Для каждого файла:**
- Запусти `pytest <file> -v` после написания
- Если тесты падают — определи: баг в тесте или баг в коде

## Anti-Patterns to REJECT

- Тесты, которые мокают {{ЗАПОЛНИТЬ: ваши ключевые внутренние модули}}
- Тесты, которые проверяют только "нет исключения" без assert на поведение
- Тесты с хардкоженными данными без объяснения ПОЧЕМУ такие данные
- Тесты, которые ломаются при переименовании приватных полей/методов
- Тесты-дубликаты с минимальными отличиями (→ parametrize)
- time.sleep() вместо condition-based waiting
- Тесты, зависящие от порядка выполнения других тестов

Этот шаблон работает для любого крупного Python-проекта — от web API до CLI-инструментов, от data pipeline до game-серверов. Универсальные навыки (1, 2, 3) переносятся без изменений; навык-оркестратор (4) заполняется за 15–30 минут под конкретный домен.
